# Middleware, Context, and Request Lifecycles

**Fast overview:** a real handler needs logging, panic recovery, auth checks, and deadlines wrapped uniformly around every route — not copy-pasted into each one. Go gets all of that from one trick, applied repeatedly: a function that takes a `Handler` and returns a `Handler`. This chapter builds that trick, then goes deep on `context.Context`, the mechanism that carries cancellation, deadlines, and request-scoped data through an entire call tree — the concrete, working payoff of the `context` package Chapter 14 introduced for goroutine cancellation in general.

## Middleware is just composition

A middleware is a function with this shape:

```go
type Middleware func(http.Handler) http.Handler
```

It takes a `Handler`, wraps it, and returns a new `Handler`. Because `Handler` is Chapter 24's one-method interface, "wrapping" just means constructing a new `HandlerFunc` that does something, calls the original handler, then maybe does something else:

```go
func Logging(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        next.ServeHTTP(w, r)
        log.Printf("%s %s %v", r.Method, r.URL.Path, time.Since(start))
    })
}
```

Stack several of these and you get a pipeline, applied outside-in on the way through and inside-out on the way back:

```go
handler := Logging(Recovery(Auth(mux)))
```

A request hits `Logging` first, which calls `Recovery`, which calls `Auth`, which finally calls `mux`. Each layer can act before calling `next`, after, or both — and any layer can choose not to call `next` at all (an `Auth` middleware that rejects a request writes a `401` and returns, never reaching the real handler). This is the decorator pattern, and it's the entire architecture of every production Go HTTP service, whether it's hand-rolled like this or provided by a router library's `Use()` method.

## Recovery middleware: Chapter 18's payoff

Chapter 18 established that `recover()` only does anything inside a `defer`, and that the right place to recover from a panic is at a natural boundary — the top of a goroutine, or here, the top of a request. Without one, a single panicking handler (a nil map write, an out-of-bounds slice index on malformed input) takes down the *entire server process*, not just that one request, because Go's HTTP server runs each request in its own goroutine and an unrecovered panic in any goroutine crashes the program. Recovery middleware turns that into what it should be — one failed request, logged, returning `500` — instead of an outage:

```go
func Recovery(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {
            if err := recover(); err != nil {
                log.Printf("panic: %v\n%s", err, debug.Stack())
                w.WriteHeader(http.StatusInternalServerError)
            }
        }()
        next.ServeHTTP(w, r)
    })
}
```

*Connect the dot:* this is exactly Chapter 18's rule made concrete — `recover` lives in a `defer` at the outermost frame of the unit of work you want to survive a panic, and here that unit of work is "one HTTP request."

## context.Context: the thread through everything

Every incoming request already carries a `Context`, reachable via `r.Context()`. The standard library wires it up for you: that context is cancelled automatically the moment the client disconnects or the underlying connection closes, so any work you do on the server's behalf can watch `ctx.Done()` and stop early instead of continuing to burn CPU and database connections for a client that already gave up.

Two things you do with that context constantly:

**Add a deadline for downstream work.** A handler that calls a database or an upstream API should almost never do so with an unbounded context — a single slow dependency would otherwise be able to hang that request (and the goroutine serving it) forever:

```go
func getPost(w http.ResponseWriter, r *http.Request) {
    ctx, cancel := context.WithTimeout(r.Context(), 3*time.Second)
    defer cancel()

    post, err := db.FetchPost(ctx, r.PathValue("id"))
    if err != nil {
        http.Error(w, "failed to fetch post", http.StatusInternalServerError)
        return
    }
    json.NewEncoder(w).Encode(post)
}
```

`context.WithTimeout` derives a *child* context from the request's context — it inherits the parent's cancellation (if the client disconnects, this child is cancelled too) and adds its own upper bound on top. `defer cancel()` releases the timer's resources the moment the function returns, whether that's because the query finished or because the timeout fired; forgetting it is a small but real resource leak in a long-running server.

**Carry request-scoped values.** A request ID generated by the outermost middleware, or an authenticated user attached by an `Auth` middleware, needs to reach handlers and logging calls several layers deeper without being threaded through every function signature by hand. `context.WithValue` does this — but with two firm rules that are violated constantly by beginners and worth stating bluntly:

- **Use an unexported custom key type, never a bare string.** A `context.WithValue(ctx, "userID", id)` risks collisions with any other package that also happens to use the string `"userID"` as a key — two unrelated packages can silently stomp each other's values. The idiomatic fix is a private type so no other package can construct a colliding key even by accident:

```go
type ctxKey int

const userIDKey ctxKey = 0

func WithUserID(ctx context.Context, id string) context.Context {
    return context.WithValue(ctx, userIDKey, id)
}

func UserIDFrom(ctx context.Context) (string, bool) {
    id, ok := ctx.Value(userIDKey).(string)
    return id, ok
}
```

- **Context values are for cross-cutting request metadata, not for parameters your function actually needs to do its job.** A request ID for log correlation, an authenticated user, a trace span — those belong in context because they travel *sideways*, through logging and middleware that has no other way to reach them, not because the function that uses them structurally needs them as an argument. A required input like "which post ID to fetch" belongs as an explicit function parameter, always. Smuggling ordinary arguments through context makes function signatures lie about what they need and makes code impossible to call correctly without reading its implementation — this is one of the most commonly violated rules in real Go codebases, and it's worth holding the line on.

## Tying request IDs to logging

Putting the two ideas from this chapter together is what makes production logs usable. A middleware generates a request ID, stores it on the context, and every subsequent log line — in the handler, in a database wrapper, in an error path several calls deep — pulls it back out and includes it, so every log line from one request can be grepped together:

```go
func RequestID(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        id := uuid.NewString()
        ctx := context.WithValue(r.Context(), requestIDKey, id)
        w.Header().Set("X-Request-Id", id)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}
```

`r.WithContext(ctx)` is the piece that's easy to forget — it returns a *shallow copy* of the request with the new context attached, and that copy, not the original `r`, is what must be passed to `next.ServeHTTP`. This pattern — a request ID attached at the edge, structured logging reading it back out everywhere downstream — is picked up again in full in Chapter 30's observability chapter, where it becomes the backbone of correlating logs, metrics, and traces across an entire distributed request.
