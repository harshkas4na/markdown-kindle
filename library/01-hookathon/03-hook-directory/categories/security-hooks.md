# Security Hooks

**63 hook projects.**

## The problem, explained simply

Because Uniswap v4 pools are permissionless — anyone can create a pool for any token pair, and anyone can attach a hook — the platform inherits every scam and exploit pattern that already exists in permissionless token launches, plus new ones specific to programmable pools:

- **Rug pulls** — a token creator seeds a pool with liquidity, waits for buyers, then pulls all the liquidity out (or dumps their own huge token holding), leaving buyers with a worthless token and no way to sell.
- **Malicious or buggy hooks** — because a hook runs arbitrary code around every swap, a malicious hook author can, in principle, do things a normal AMM never could (e.g. trap funds, redirect fees, selectively block certain users).
- **Sniping** — bots that detect a pool's exact creation or first-liquidity-add transaction and buy before real users can, then dump on them.
- **Flash-loan-driven manipulation** — an attacker borrows a huge amount of capital for a single transaction to distort a pool's price or drain funds, then repays the loan in the same transaction, leaving no lasting trace besides the damage.

## Why so many teams attack this

Security incidents are the single biggest reputational and financial risk in DeFi, and are highly visible (rug pulls and exploits get widely reported), so building visible protective mechanisms is a natural, relatable hackathon pitch: "trust me, my hook stops the bad thing that just happened to some other project."

## Common approaches seen in this dataset

1. **Circuit breakers** — inspired directly by traditional stock-exchange circuit breakers (which halt trading after an extreme price move to give the market time to reassess): pause or restrict trading in a pool if the price moves too far too fast in one block/transaction.
2. **Anti-sniping protections** — detect and block or penalize transactions that try to buy immediately at pool creation, often by adding an initial cooldown or a minimum liquidity/lockup requirement before normal trading opens.
3. **Insurance mechanisms** — collect a small insurance premium from swaps or LPs, and pay out compensation automatically if a defined bad event (a rug pull, a depeg, a hook exploit) is detected.
4. **Liquidity locking / fair-launch enforcement** — require that a meaningful share of a new token's liquidity is locked for a minimum period before allowing trading, directly reducing rug-pull risk for meme/community tokens.
5. **Hook auditing/composability safety** — frameworks or adapters meant to make it safer to combine multiple third-party hooks on one pool without one hook interfering with or exploiting another.

## Prior art / reference material

- NASDAQ and other traditional-market circuit-breaker rules — the explicit inspiration named by at least one project (uniguard.exchange) in this dataset for pausing pathological trading.
- Historical rug pull and DeFi exploit case studies (widely documented across the industry, e.g. via RugDoc, DeFi Safety, and post-mortem write-ups) — the general threat model this whole category defends against.
- Standard smart-contract security practices (reentrancy guards, flash-loan-resistant design) as codified in widely used references like the OpenZeppelin contracts library and Trail of Bits' smart contract security guidelines.
